# sqlc

sqlc generates type-safe Go code from SQL query files. You write plain SQL; sqlc validates it against the schema and emits idiomatic Go functions and structs.

## How it works

1. Write a query in `db/queries/` with a sqlc annotation:
   ```sql
   -- name: GetSignupsOpen :one
   SELECT value FROM telive.settings WHERE key = 'signups_open';
   ```
2. Run `sqlc generate`. It reads the query files and `db/schema.sql`, validates them, and emits Go code into `db/sqlcdb/`.
3. Commit both — the hand-written `.sql` source and the generated `.go` output.

`db/sqlcdb/` files have a `// Code generated by sqlc. DO NOT EDIT.` header. The `.sql` files are the source of truth.

## Value proposition

### Compile-time query validation

`sqlc generate` validates queries against `db/schema.sql` at code generation time. A typo like `SELECT nme FROM telive.signups` fails immediately with a clear error. With raw `db.QueryRowContext`, the same mistake only surfaces when that code path runs in production — which could be infrequently.

### No string construction

Dynamic SQL built from concatenated strings can silently produce malformed queries. sqlc queries are fixed strings with `$1`/`$2` bind variables, never constructed at runtime. Once a query is in a `.sql` file and generates without error, that specific malformation class is eliminated.

The one place in this project where sqlc doesn't apply is the dynamic `IN (...)` clause in `Add()`, which builds a variable-length placeholder list at runtime — that stays as raw SQL by design.

### Type-safe result structs

Manual scanning requires columns in the exact right order. A mistake is silent or panics at runtime:

```go
// Raw SQL — column order matters; wrong order silently produces bad data
rows.Scan(&entryID, &name, &songID, &title, &artist, &tabUrl, &performed, &timesOnStage)
```

sqlc generates a named struct and scans into it:

```go
// Generated — fields are named; adding a column and forgetting the scan is a compile error
rows.Scan(&i.Singer, &i.ID, &i.Title, &i.Artist, &i.TabUrl)
```

### Less boilerplate

Every raw query requires the same ~8 lines of infrastructure: `defer rows.Close()`, the `rows.Next()` loop, and `rows.Err()` after the loop. Forgetting `rows.Err()` silently swallows mid-iteration errors. sqlc generates this pattern once, correctly, for every query. See `db/sqlcdb/songs.sql.go` for a representative example.

## Workflow

```
# After editing a .sql file or db/schema.sql:
sqlc generate

# Or via the justfile:
just generate
```

`db/schema.sql` must stay in sync with Liquibase migrations — sqlc reads it as the source of truth for table shapes. The CI liquibase workflow enforces that migrations and `schema.sql` are updated in the same commit.
